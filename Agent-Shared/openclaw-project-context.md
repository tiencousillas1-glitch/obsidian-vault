## 8. Orthodontic clinic business economics

Useful niche facts:

- A single-location orthodontic practice often generates approximately **$1.5M–$2.5M per year** in revenue.
- High-performing practices may exceed **$3M annually**.
- Expenses commonly consume around **65–70%** of revenue.
- Typical cost categories include staff, lab/Invisalign fees, rent, utilities, marketing, software, phones, and payment processing.
- A typical orthodontic team may have **5–8 employees per orthodontist**, including front desk, a treatment coordinator, and assistants.
- Net margin can be around **30–35% before taxes**.
- A well-run single practice may produce around **$400K–$600K owner take-home per year**.

Why this matters:

A missed new-patient consultation is not a small loss. Orthodontic treatment can be high-ticket and long-term. Losing one motivated patient because the phone was not answered can mean losing a meaningful revenue opportunity.

## 23. OpenClaw troubleshooting principles

When troubleshooting OpenClaw, follow this order:

1. Identify the exact failing surface: Telegram, Discord, web dashboard, VPS, Docker container, model provider, auth profile, port access, or Tailscale.
2. Do not assume provider is OpenAI.
3. Check container status.
4. Check logs.
5. Check whether service is bound to `127.0.0.1` or `0.0.0.0`.
6. Check Tailscale IP access separately from local loopback access.
7. Check `.env` and auth profiles.
8. Check Telegram bot token and webhook/polling mode.
9. Check allowlist/group policy.
10. Restart only the necessary service/container.
11. Avoid destructive resets unless there is a backup or clear reason.

Useful command categories:

```bash
docker ps
docker logs <container_name_or_id> --tail=100
docker compose ps
docker compose logs -f
tailscale status
curl http://127.0.0.1:18789
curl http://127.0.0.1:18791
```

The agent should translate errors into direct actions, not overwhelm the user.

## 26. Handoff and heartbeat concepts

The user wants to work on "handoff" and "heartbeat" structures for agents.

### Handoff

A handoff is when one agent passes context, task state, or responsibility to another agent or system.

A useful handoff should include:

- Current objective.
- What has already been done.
- What is blocked.
- Required next action.
- Relevant files/links/credentials status without exposing secrets.
- Decision needed.
- Owner of next step.
- Deadline or urgency.

Handoff format:

```markdown
## Handoff: [Task/Area]

**Current goal:**
**Status:**
**Completed:**
**Blocked by:**
**Next action:**
**Relevant context:**
**Do not change:**
**Definition of done:**
```

### Heartbeat

A heartbeat is a recurring status update that keeps the system alive and prevents silent failure.

A heartbeat should report:

- Is the system running?
- What changed since last check?
- Any errors?
- Any stuck tasks?
- Any revenue-moving action completed?
- What is the next action?

Heartbeat format:

```markdown
## Heartbeat: [System/Agent]

**Status:** Running / Blocked / Needs attention
**Last action completed:**
**Current blocker:**
**Risk:**
**Next revenue-moving action:**
**Human input needed:** Yes/No
```
